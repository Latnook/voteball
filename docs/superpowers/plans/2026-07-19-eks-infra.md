# EKS Infra (Terraform) Implementation Plan — Plan 2

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Stand up the AWS foundation for the Voteball EKS migration in a new, independent Terraform stack — a dedicated VPC, an EKS cluster + managed node group, OIDC/IRSA roles for the app, and the ECR/ACM/S3/Secrets Manager/SNS resources the app will consume — verified by `kubectl get nodes` against the live cluster.

**Architecture:** A **separate** stack at `terraform/` with its own **local** state, parallel to the existing k3s `terraform/` stack (they share nothing). The **VPC and EKS clusters use the official `terraform-aws-modules` community modules** (battle-tested OIDC/node-group wiring, least code to get wrong); everything else — **IRSA IAM policies, ECR, ACM, S3, Secrets Manager, SNS** — is hand-rolled in the repo's existing small, heavily-commented style, because the least-privilege IRSA JSON is the security centerpiece the course rubric grades hardest and must stay transparent.

**Tech Stack:** Terraform ≥ 1.5, `hashicorp/aws ~> 5.0` (NOT the k3s stack's `~> 6.0` — the EKS module v20 caps AWS at `< 6.0.0`; independent stack, harmless skew), `terraform-aws-modules/vpc/aws ~> 5.0`, `terraform-aws-modules/eks/aws ~> 20.0`, region `il-central-1`, AWS account `590183895228`.

## Global Constraints

*Every task's requirements implicitly include this section. Values are copied verbatim from the approved design (`project_eks_migration_plan` memory + `docs/plan.md` global constraints).*

- **Region** `il-central-1`; **AZs** `il-central-1a` + `il-central-1b` (EKS needs ≥2; ALB needs subnets in ≥2 AZs). **Account** `590183895228`.
- **Resource name prefix** `voteball`; single environment; no dev/prod split.
- **This stack is its own root module** (`terraform/`) with independent local state — a **separate folder, deliberately NOT a Terraform workspace on the k3s config** (workspaces share one configuration; k3s and EKS are different infra, so a shared config would need `count`-toggle-per-resource anti-patterns). Labeled the `dev` environment via the provider's `default_tags { Environment = "dev" }`, which auto-stamps every taggable resource including the community modules'.
- **Dedicated VPC** `10.0.0.0/16` (NOT the default VPC the k3s stack uses). Six subnets: public `10.0.0.0/20`(1a) + `10.0.16.0/20`(1b); private `10.0.32.0/20`(1a) + `10.0.48.0/20`(1b); database `10.0.64.0/24`(1a) + `10.0.65.0/24`(1b). **Single NAT gateway** (cost; documented trade-off = 1a dependency).
- **Kubernetes namespace** the app will use is `devops-app` — IRSA trust policies reference `system:serviceaccount:devops-app:<sa>` (the namespace/SAs themselves are created later by the Helm chart in Plan 3; the roles can reference them by name before they exist).
- **IRSA least privilege:** worker SA → `sns:Publish` (topic ARN) + `s3:PutObject` on `snapshots/*` only; backup SA → `s3:PutObject` on `backups/*` only, a **separate** role; backend/frontend SAs get **no** AWS role at all. **No workload gets cluster-admin.**
- **Pin the EKS version** (`var.cluster_version`) — extended support is $0.50/hr vs $0.10/hr standard (5× cliff); track the standard-support end date.
- **ECR image scanning on push** enabled (feeds the rubric's "image scanned?" line); no `latest` tags — images are pushed with git-SHA tags by the deploy pipeline (later plan).
- **Secret values NEVER enter `terraform.tfstate`:** Terraform creates the Secrets Manager container + a placeholder version with `lifecycle { ignore_changes = [secret_string] }`; a human seeds real values once via `aws secretsmanager put-secret-value` (documented manual op).
- **Stateful-op safety:** run all `terraform` commands from `terraform/` in the main working tree, never a git worktree (state lives where the command runs — see the `never-run-stateful-infra-ops-in-a-worktree` note). Stream long applies live.
- **Cost discipline:** `terraform apply` here creates billed resources (EKS control plane ~$73/mo + NAT ~$37/mo + Spot nodes ~$50/mo). Only the final task applies; earlier tasks stop at `validate`/`plan`. Tear down between sessions.
- **Commit and push to `master` as each task completes** (repo standing order). Plain imperative commit messages. Never force-push.

**Deviation from the design — accept or reject before executing:** the design specified a *self-managed* node group with a mixed-instances ASG policy (Spot-primary, On-Demand fallback). This plan instead uses a **managed node group with `capacity_type = "SPOT"` and diversified instance types** (`t3.large`, `t3a.large`) — the community module's idiomatic path, far less code. It keeps Spot resilience through instance-type diversity + (Plan 2b) Cluster Autoscaler, but trades away automatic On-Demand fallback if all Spot pools are exhausted. Acceptable for a demo cluster; documented as a trade-off. If you want the true fallback, this becomes a `self_managed_node_groups` block instead.

**Pre-flight:** confirm AWS creds work for account `590183895228`: `aws sts get-caller-identity` → account `590183895228`.

### Task 2: Dedicated VPC (community module)

Create the EKS VPC via `terraform-aws-modules/vpc/aws` — six subnets across two AZs, a single NAT gateway, and the subnet tags EKS/ALB use for discovery.

**Files:**
- Create: `terraform/vpc.tf`

**Interfaces:**
- Consumes: `var.vpc_cidr`, `var.azs`, `var.cluster_name`.
- Produces: `module.vpc.vpc_id`, `module.vpc.private_subnets` (node/pod placement), `module.vpc.public_subnets` (ALB), `module.vpc.database_subnets` (RDS later).

- [ ] **Step 1: Write `terraform/vpc.tf`**

```hcl
# Dedicated VPC for EKS (the k3s stack uses the default VPC; this one is isolated).
# Public subnets host the ALB + NAT GW; private subnets host nodes/pods; database subnets are
# isolated (no NAT/IGW route) for a future RDS move. Single NAT GW to save ~$35/mo -- trade-off is
# an AZ-1a dependency for egress, acceptable for a course project.
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "~> 5.0"

  name = "${var.cluster_name}-eks-vpc"
  cidr = var.vpc_cidr
  azs  = var.azs

  public_subnets   = ["10.0.0.0/20", "10.0.16.0/20"]
  private_subnets  = ["10.0.32.0/20", "10.0.48.0/20"]
  database_subnets = ["10.0.64.0/24", "10.0.65.0/24"]

  enable_nat_gateway   = true
  single_nat_gateway   = true
  enable_dns_hostnames = true
  enable_dns_support   = true

  # EKS/ALB subnet discovery tags: public subnets host internet-facing LBs, private host internal.
  public_subnet_tags  = { "kubernetes.io/role/elb" = "1" }
  private_subnet_tags = { "kubernetes.io/role/internal-elb" = "1" }

  # Project/Environment come from the provider default_tags; only the functional EKS discovery tag
  # is set here.
  tags = {
    "kubernetes.io/cluster/${var.cluster_name}" = "shared"
  }
}
```

```hcl
# the Terraform caller cluster-admin via an EKS access entry, so `kubectl get nodes` works right

  version = "~> 20.0"
```

- [ ] **Step 1: Write `terraform/ecr.tf`**

```hcl
# One repo per own-image. scan_on_push turns on ECR's built-in vulnerability scan (the rubric's
# "is the image scanned?" line). Untagged images expire after 14 days to bound storage cost.
locals {
  ecr_repos = ["backend", "worker", "nginx"]
}

resource "aws_ecr_repository" "app" {
  for_each             = toset(local.ecr_repos)
  name                 = "${var.cluster_name}-${each.key}"
  image_tag_mutability = "IMMUTABLE" # git-SHA tags are unique; immutability prevents silent overwrite

  image_scanning_configuration {
    scan_on_push = true
  }
}

resource "aws_ecr_lifecycle_policy" "app" {
  for_each   = aws_ecr_repository.app
  repository = each.value.name

  policy = jsonencode({
    rules = [{
      rulePriority = 1
      description  = "Expire untagged images after 14 days"
      selection = {
        tagStatus   = "untagged"
        countType   = "sinceImagePushed"
        countUnit   = "days"
        countNumber = 14
      }
      action = { type = "expire" }
    }]
  })
}
```

- [ ] **Step 2: Write `terraform/acm.tf`**

```hcl
# ACM cert for the app FQDN, DNS-validated by writing the validation CNAME into the existing zone.
# This replaces the k3s certbot/Let's-Encrypt mechanism (and its rate limits) -- the ALB (Plan 3)
# references this cert's ARN.
resource "aws_acm_certificate" "app" {
  domain_name       = var.app_domain
  validation_method = "DNS"

  lifecycle {
    create_before_destroy = true
  }
}

resource "aws_route53_record" "acm_validation" {
  for_each = {
    for dvo in aws_acm_certificate.app.domain_validation_options : dvo.domain_name => {
      name   = dvo.resource_record_name
      type   = dvo.resource_record_type
      record = dvo.resource_record_value
    }
  }

  zone_id         = data.aws_route53_zone.primary.zone_id
  name            = each.value.name
  type            = each.value.type
  records         = [each.value.record]
  ttl             = 60
  allow_overwrite = true
}

resource "aws_acm_certificate_validation" "app" {
  certificate_arn         = aws_acm_certificate.app.arn
  validation_record_fqdns = [for r in aws_route53_record.acm_validation : r.fqdn]
}
```

### Task 6: Hand-rolled IRSA roles (worker + backup)

The least-privilege centerpiece. Two roles, each federating the cluster OIDC provider to one specific service account, with hand-written policy JSON — worker gets SNS-publish + S3-write to `snapshots/` only; backup gets S3-write to `backups/` only. Backend/frontend get no role.

**Files:**
- Create: `terraform/irsa.tf`

**Interfaces:**
- Consumes: `module.eks.oidc_provider` (issuer host/path), `module.eks.oidc_provider_arn`, `aws_sns_topic.notifications.arn`, `aws_s3_bucket.rollups.arn`.
- Produces: `aws_iam_role.worker.arn`, `aws_iam_role.backup.arn` — annotated onto the `devops-app:worker` / `devops-app:backup` service accounts by the Helm chart (Plan 3).

- [ ] **Step 1: Write `terraform/irsa.tf`**

```hcl
# IRSA = IAM Roles for Service Accounts. Each role's trust policy federates the cluster's OIDC
# provider to ONE specific service account (sub) with audience sts.amazonaws.com. This is the
# concrete least-privilege story: two workloads touching the same bucket under DIFFERENT prefixes
# with DIFFERENT roles, and backend/frontend get no role at all.

# ---- worker: milestone alerts (SNS) + rollup snapshots (S3 snapshots/ only) ----
data "aws_iam_policy_document" "worker_trust" {
  statement {
    effect  = "Allow"
    actions = ["sts:AssumeRoleWithWebIdentity"]
    principals {
      type        = "Federated"
      identifiers = [module.eks.oidc_provider_arn]
    }
    condition {
      test     = "StringEquals"
      variable = "${module.eks.oidc_provider}:sub"
      values   = ["system:serviceaccount:devops-app:worker"]
    }
    condition {
      test     = "StringEquals"
      variable = "${module.eks.oidc_provider}:aud"
      values   = ["sts.amazonaws.com"]
    }
  }
}

data "aws_iam_policy_document" "worker_permissions" {
  statement {
    sid       = "PublishMilestones"
    effect    = "Allow"
    actions   = ["sns:Publish"]
    resources = [aws_sns_topic.notifications.arn]
  }
  statement {
    sid       = "WriteSnapshots"
    effect    = "Allow"
    actions   = ["s3:PutObject"]
    resources = ["${aws_s3_bucket.rollups.arn}/snapshots/*"] # write-only, snapshots/ prefix only
  }
}

resource "aws_iam_role" "worker" {
  name               = "${var.cluster_name}-worker-irsa"
  assume_role_policy = data.aws_iam_policy_document.worker_trust.json
}

resource "aws_iam_role_policy" "worker" {
  name   = "${var.cluster_name}-worker-permissions"
  role   = aws_iam_role.worker.id
  policy = data.aws_iam_policy_document.worker_permissions.json
}

# ---- backup CronJob: nightly DB/results dump (S3 backups/ only) -- its OWN role, no SNS ----
data "aws_iam_policy_document" "backup_trust" {
  statement {
    effect  = "Allow"
    actions = ["sts:AssumeRoleWithWebIdentity"]
    principals {
      type        = "Federated"
      identifiers = [module.eks.oidc_provider_arn]
    }
    condition {
      test     = "StringEquals"
      variable = "${module.eks.oidc_provider}:sub"
      values   = ["system:serviceaccount:devops-app:backup"]
    }
    condition {
      test     = "StringEquals"
      variable = "${module.eks.oidc_provider}:aud"
      values   = ["sts.amazonaws.com"]
    }
  }
}

data "aws_iam_policy_document" "backup_permissions" {
  statement {
    sid       = "WriteNightlyBackups"
    effect    = "Allow"
    actions   = ["s3:PutObject"]
    resources = ["${aws_s3_bucket.rollups.arn}/backups/*"] # write-only, backups/ prefix only, no SNS
  }
}

resource "aws_iam_role" "backup" {
  name               = "${var.cluster_name}-backup-irsa"
  assume_role_policy = data.aws_iam_policy_document.backup_trust.json
}

resource "aws_iam_role_policy" "backup" {
  name   = "${var.cluster_name}-backup-permissions"
  role   = aws_iam_role.backup.id
  policy = data.aws_iam_policy_document.backup_permissions.json
}
```
